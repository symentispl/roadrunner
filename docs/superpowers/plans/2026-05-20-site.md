> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add an Antora-based documentation site to the Roadrunner repo, built from `site/` and published to GitHub Pages on every release.

**Architecture:** AsciiDoc source files live under `site/modules/ROOT/pages/`. `antora-playbook.yml` at the repo root drives the build, outputting to `target/site`. The existing `release.yml` GitHub Actions workflow gains three steps after JReleaser: set up Node.js, run `npx antora`, deploy to the `gh-pages` branch via `peaceiris/actions-gh-pages`.

**Tech Stack:** Antora 3.x (via `npx`), AsciiDoc, GitHub Pages, GitHub Actions (`peaceiris/actions-gh-pages@v4`)

= Roadrunner

image:https://codecov.io/gh/symentispl/roadrunner/graph/badge.svg?token=37S96CL3YR[Codecov,link=https://codecov.io/gh/symentispl/roadrunner]

Roadrunner is a Java load generator built on virtual threads (JDK 25+).
It is protocol-agnostic, low-overhead, and designed for continuous performance management.

== Why Roadrunner?

* *Protocol-agnostic* — built-in HTTP, JDBC, and VM protocols; add your own via the plugin API
* *Low overhead* — virtual-thread-per-user model with coordinated omission correction
* *Simple* — no scripting, no complex scenarios; point it at a target and go
* *Extensible reporting* — console summary and HTML report with HDR histogram charts

== Quick start

[source,bash]
----
./mvnw verify
roadrunner -c 50 -n 500 vm --sleep-time 10
----

See xref:getting-started.adoc[Getting Started] for full instructions.

= Getting Started

== Prerequisites

* https://adoptium.net/temurin/releases/?version=25[JDK 25 (Temurin)] or later
* Git

NOTE: Roadrunner is currently tested and supported on Linux only.

== Build from source

[source,bash]
----
git clone https://github.com/symentispl/roadrunner.git
cd roadrunner
./mvnw verify
----

The runnable binary is produced at:

----
roadrunner-app/target/maven-jlink/default/bin/roadrunner
----

Add it to your `PATH` or run it directly:

[source,bash]
----
export PATH=$PWD/roadrunner-app/target/maven-jlink/default/bin:$PATH
----

== Run your first test

The `vm` protocol sleeps for a fixed number of milliseconds per request.
It is useful for verifying Roadrunner overhead and as a baseline.

[source,bash]
----
roadrunner -c 50 -n 500 vm --sleep-time 10
----

This sends 500 requests using 50 concurrent users, each sleeping 10 ms.

== Run an HTTP test

The `ab` protocol benchmarks an HTTP endpoint using the same CLI as
https://httpd.apache.org/docs/2.4/programs/ab.html[Apache HTTP server benchmarking tool]:

[source,bash]
----
roadrunner -c 50 -n 500 ab http://localhost:8080/api/health
----

== Reading the output

After each run, Roadrunner prints a console summary and writes an HTML report to the output directory (default: a timestamped directory in the current working directory).

The HTML report includes:

* Throughput (requests/second)
* Latency percentiles (p50, p90, p99, p999, max)
* HDR histogram chart
* Coordinated-omission-corrected latency (when pause detectors are enabled)

See xref:configuration.adoc[Configuration] for all available options.
```

= Configuration

Roadrunner is invoked as:

[source]
----
roadrunner [OPTIONS] <protocol> [PROTOCOL_OPTIONS]
----

== Load model

Exactly one load model must be chosen per run.

=== Closed-world model

All concurrent users loop continuously until the total request count is reached.

[cols="1,3"]
|===
| Option | Description

| `-c <n>`
| Number of concurrent users (required)

| `-n <n>`
| Total number of requests to make (required)
|===

Example:

[source,bash]
----
roadrunner -c 50 -n 1000 vm --sleep-time 5
----

=== Open-world model

New requests arrive at a fixed rate regardless of how long in-flight requests take.
This models real-world arrival patterns and avoids coordinated omission.

[cols="1,3"]
|===
| Option | Description

| `--rate <n>`
| Target request rate in requests per second (required)

| `--duration <d>`
| Test duration, e.g. `30s`, `5m`, `2h` (required)
|===

Example:

[source,bash]
----
roadrunner --rate 100 --duration 30s ab http://localhost:8080/
----

== Output and reporting

[cols="1,1,3"]
|===
| Option | Default | Description

| `-s <path>`
| timestamped dir
| Output directory for results CSV and report

| `-r <format>`
| `console`
| Report format. Built-in values: `console`, `html`
|===

HTML report example:

[source,bash]
----
roadrunner -c 10 -n 200 -r html ab http://localhost:8080/
----

== Latency correction

[cols="1,3"]
|===
| Option | Description

| `--pause-detectors vt,jvm`
| Enable pause-corrected latency recording. Accepts `vt` (virtual-thread scheduler pauses), `jvm` (GC/JVM pauses), or `vt,jvm` for both. Only supported with the open-world model.

| `--raw-latency`
| Force reports to use the per-event CSV histogram even when a pause-corrected `latency.hgrm` is present.
|===

[#parameter-sources]
== Parameter sources

Feed a dynamic set of parameters into each request (e.g. different SQL queries or URLs per user):

[cols="1,3"]
|===
| Option | Description

| `--parameters-source <type>:<key>=<value>`
| Parameter source in `type:key=value` format. Built-in type: `csv:file=<path>`
|===

Example — pass a different query per request from a CSV file:

[source,bash]
----
roadrunner -c 10 -n 100 --parameters-source csv:file=queries.csv jdbc \
  --url jdbc:postgresql://localhost/mydb \
  --username app --password secret \
  --driver /opt/drivers/postgresql.jar \
  "SELECT 1"
----
```

= Samplers

Roadrunner is sampler-agnostic. Each sampler is a pluggable implementation that
executes one request per virtual-thread iteration and returns a `ProtocolResponse`
with nanosecond-precision timestamps.

== Built-in samplers

The following samplers are bundled with the standard distribution:

[cols="1,3"]
|===
| Sampler | Description

| xref:vm.adoc[vm]
| In-VM baseline — sleeps for a fixed number of milliseconds. Use this to measure Roadrunner's own overhead.

| xref:ab.adoc[ab]
| HTTP benchmarking, API-compatible with Apache's `ab` tool.

| xref:zero.adoc[zero]
| Zero-latency no-op. Useful for measuring the absolute ceiling throughput of the engine.
|===

== Plugin samplers

The following samplers are distributed as separate plugin JARs.
Install them by placing the JAR under `~/.roadrunner/plugins/protocols/`.

[cols="1,3"]
|===
| Sampler | Description

| xref:jdbc.adoc[jdbc]
| Executes a SQL query against any JDBC-compatible database.

| xref:neo4j.adoc[neo4j]
| Executes a Cypher query against a Neo4j database.
|===

== Writing your own sampler

Implement `io.roadrunner.api.samplers.SamplerProvider` from the `roadrunner-samplers-spi` module,
annotate it with `@Command` (picocli) for CLI options, and register it via
`META-INF/services/io.roadrunner.api.samplers.SamplerProvider` or a `module-info.java` `provides` directive.

See `VmSamplerProvider` and `AbSamplerProvider` in the source for reference implementations.
```

= VM Protocol

The `vm` protocol is an in-process baseline sampler. Each request sleeps for a fixed
number of milliseconds on a virtual thread, then returns immediately. Its purpose is
to measure Roadrunner's own scheduling and recording overhead without any real I/O.

== Usage

[source,bash]
----
roadrunner -c <concurrency> -n <requests> vm --sleep-time <ms>
----

== Options

[cols="1,1,3"]
|===
| Option | Required | Description

| `--sleep-time <ms>`
| Yes
| Sleep duration in milliseconds per request
|===

== Example

[source,bash]
----
roadrunner -c 50 -n 500 vm --sleep-time 10
----

500 requests, 50 concurrent users, each sleeping 10 ms per request.
Expected throughput ≈ 50 / 0.010 = 5000 req/s (minus overhead).

== When to use it

* Establish a latency floor before testing real protocols
* Verify that observed latencies are not inflated by Roadrunner overhead
* Smoke-test that the binary is installed correctly
```

= Zero Protocol

The `zero` protocol is a no-op sampler. Each request returns immediately with zero
latency and no I/O. It measures the absolute maximum throughput the Roadrunner
engine can sustain — the ceiling beyond which no protocol optimisation can go.

== Usage

[source,bash]
----
roadrunner -c <concurrency> -n <requests> zero
----

== Options

The `zero` protocol has no protocol-specific options.

== Example

[source,bash]
----
roadrunner -c 100 -n 1000000 zero
----

Runs one million requests across 100 concurrent users with zero work per request.
The resulting throughput is Roadrunner's engine overhead baseline.

== When to use it

* Determine the engine's maximum achievable request rate on your hardware
* Confirm that a protocol under test has room to improve before optimising Roadrunner itself
```

= JDBC Protocol

The `jdbc` protocol executes a SQL query against any JDBC-compatible database.
It is distributed as a plugin JAR, not bundled with the standard distribution.

== Installation

Download the `roadrunner-sampler-jdbc-<version>.jar` from the
https://github.com/symentispl/roadrunner/releases[GitHub Releases] page and place it
under `~/.roadrunner/plugins/protocols/`:

[source,bash]
----
mkdir -p ~/.roadrunner/plugins/protocols
cp roadrunner-sampler-jdbc-*.jar ~/.roadrunner/plugins/protocols/
----

You also need the JDBC driver JAR for your database. Pass its path with `--driver`.

== Usage

[source,bash]
----
roadrunner -c <concurrency> -n <requests> jdbc [OPTIONS] "<sql>"
----

== Arguments

[cols="1,3"]
|===
| Argument | Description

| `<sql>`
| SQL query to execute per request (required, positional)
|===

== Options

[cols="1,1,3"]
|===
| Option | Required | Description

| `--url <jdbc-url>`
| Yes
| JDBC URL, e.g. `jdbc:postgresql://localhost/mydb`

| `--username <user>`
| Yes
| Database username

| `--password <pass>`
| Yes
| Database password

| `--driver <path>`
| Yes
| Path to the JDBC driver JAR file

| `--driver-class <fqcn>`
| No
| Fully-qualified JDBC driver class name. Auto-detected from the JAR if omitted.

| `--pool-size <n>`
| No
| Maximum connection pool size. Defaults to the concurrency level (`-c`).
|===

== Example

[source,bash]
----
roadrunner -c 20 -n 1000 jdbc \
  --url jdbc:postgresql://localhost/benchdb \
  --username app \
  --password secret \
  --driver /opt/drivers/postgresql-42.7.3.jar \
  "SELECT id, name FROM users WHERE id = 1"
----

== Parameter sources

Combine with `--parameters-source csv:file=<path>` to execute a different query per
request using values from a CSV file.
See xref:configuration.adoc#parameter-sources[Parameter sources] for details.
```
